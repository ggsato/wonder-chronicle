# Wonder Chronicle — MVP実装ブリーフ v0.3

## 1. このドキュメントの目的
この文書は、**Wonder Chronicle** の最初の実装を始めるための共通土台です。  
用途は次の2つです。

1. **Codexレビュー用**  
   実装前に、UI構造・状態・データ構造・CSS方針の解釈ずれを減らす。
2. **初期実装の指示書**  
   MVPを一度最後まで形にするための、最小限かつ具体的な実装要件を定義する。

このプロジェクトでは、仕様書を肥大化させるより、**年表として成立する最初の実装を早く見ること**を優先する。

---

## 2. プロダクトの核

### 2.1 一言でいうと
**Wonder Chronicle は、3Wと写真から、自分だけの縦書き年表を読むためのアプリである。**

### 2.2 3Wの定義
- **Wish**: その時期に願っていたこと
- **Wonder at**: 心を動かされたこと
- **Wonder about**: 考えていたこと、問い、迷い

### 2.3 大原則
- 主役は**年表**
- 入力は**補助**
- 日本語の縦書きを中核に置く
- 「日記アプリ」ではなく「**自分史ビューア**」として設計する
- AIは文章生成者ではなく、**年表を編む編集者**として使う

---

## 3. MVPの範囲を明確にする

### 3.1 初回実装で入れるもの
初回実装では、以下を **MVPの必須範囲** とする。

- 年表画面
- 入力画面
- 詳細画面の最小版
- **巻物型**
- 粒度切替: **日 / 週 / 月**
- 時間の向き切替: **右=現在 / 右=過去**
- 起点移動: **today / latest / earliest**
- ローカルサンプルデータ
- ローカル画像アップロード
- 仮ロジックによる週・月集約
- **1日1件の記録制約**

### 3.2 初回実装では入れないもの
以下は **Phase 2** とする。

- **社史型**
- 年表示
- AI要約の本実装
- ブログ / SNSインポート
- last-view の永続化
- 1日に複数記録
- 時刻ベースの詳細ソート
- 高度な検索やフィルタ

### 3.3 なぜこの切り分けにするか
巻物型だけでも、
- 縦書き年表として成立するか
- 粒度切替が見えるか
- 向き切替が自然か
- 3Wと写真だけで時間が読めるか

を十分確認できる。  
社史型は重要だが、**「編集された時代区分」** の設計が入るため、初回実装より一段階あとで詰める。

また、**初回実装では 1日1件** に固定することで、
- 日表示 = 1日1列 = 1記録
- 詳細画面 = その期間に含まれる日記録一覧
- 代表写真 = その日または期間の代表1枚

という対応が安定する。

### 3.4 ただしUI上の見通しは残す
初回実装では社史型を実装しないが、将来追加しやすいように、
- `layoutMode` という状態名は残す
- ただしMVPでは `scroll` のみ使う
- `editorial` は未実装扱いにする

## 4. MVPで実現したい体験

### 4.1 ユーザー体験
ユーザーはアプリを開くと、まずフォームではなく**年表そのもの**を見る。  
その年表は、以下を切り替えながら読める。

- 型: MVPでは **巻物型のみ**
- 時間の向き: **右=現在 / 右=過去**
- 粒度: **日 / 週 / 月**
- 起点移動: **today / latest / earliest**

### 4.2 MVPの判断基準
以下が成立すれば、MVPは成功とみなす。

- 「これは年表だ」と一目で分かる
- カードUIに見えない
- 縦書きが成立している
- 3Wと写真だけで時間の流れが見える
- 粒度切替で見え方が変わる
- 入力が最小限で済む
- **1日1件制約のもとで、日表示・週表示・月表示の責務がぶれない**

## 5. 記録単位を固定する

### 5.1 MVPの前提
**MVPでは 1日につき 1件の JournalEntry のみ許可する。**

### 5.2 入力時の挙動
- 同じ日付に既存の記録がない場合: 新規作成
- 同じ日付に既存の記録がある場合: **新規追加ではなく更新** として扱う

### 5.3 これにより固定される意味
- **日表示**: 1日 = 1列 = 1記録
- **週表示**: その週に属する日記録の集約
- **月表示**: その月に属する日記録の集約
- **詳細画面**: その期間に属する日記録一覧

### 5.4 Phase 2以降
将来的に 1日複数記録を許す場合は、
- `createdAt`
- `sortOrder`
- 詳細画面での時系列
- 代表記録 / 代表写真選定ロジック

を追加する。  
ただしそれはMVP外とする。

## 6. 時間・日付の基準を固定する

### 6.1 タイムゾーン
- すべて **Asia/Tokyo** 前提で扱う
- 日付集約、週集約、today 判定はすべて日本時間で行う

### 6.2 週の開始曜日
- 週の開始は **月曜日**
- 1週は 月曜 00:00 から 日曜 23:59:59 まで

### 6.3 粒度別の期間定義
- **day**: 1暦日
- **week**: 月曜始まりの1週間
- **month**: 1暦月

### 6.4 空期間の扱い
MVPでは、**空の週・月は表示しない**。  
理由:
- 初回実装では時間軸の連続穴埋めより、実データの見え方を優先する
- 空期間の表現は将来追加でよい

---

## 7. 起点移動の厳密な意味

### 7.1 today
- 現在の日本時間の日付を含む期間へ移動する
- day表示: 今日の列
- week表示: 今日を含む週の列
- month表示: 今日を含む月の列
- 該当期間に記録がない場合は、**直近の過去の記録期間**へ移動する
- それも存在しなければ earliest にフォールバックする

### 7.2 latest
- 現在の粒度で見たときの、**もっとも新しい記録を含む期間**へ移動する
- 最新記録の日付を含む日/週/月が対象

### 7.3 earliest
- 現在の粒度で見たときの、**もっとも古い記録を含む期間**へ移動する

### 7.4 last-view
- 概念としては残す
- MVPではUI・永続化ともに未実装
- 将来 `localStorage` などで保持予定

---

## 8. 向き切替の実装方針を固定する

### 8.1 用語
- `recent-right`: 右が現在、左が過去
- `past-right`: 右が過去、左が現在

### 8.2 何を変えるか
向き切替時は、以下の3つを分けて考える。

1. **データ順**
2. **描画順**
3. **初期スクロール位置 / ジャンプ位置**

### 8.3 固定方針
- **データ順は常に昇順（古い→新しい）で保持する**
- **描画順で向きを表現する**
- **スクロール位置は、切替後も“同じ期間”が見えるよう補正する**

### 8.4 実装方針
- 集約後の配列は `startDate` 昇順で固定
- `recent-right` のときは、視覚的に右側が新しい期間になるよう並べる
- `past-right` のときは、視覚的に右側が古い期間になるよう並べる
- 必要に応じて描画用配列を反転する
- 反転後も「今見ている periodId」を保持し、切替後にその periodId が可視範囲へ来るようスクロール補正する

### 8.5 重要注意
- CSSだけの `transform: scaleX(-1)` で済ませない
- 文字や画像、スクロール方向の意味が崩れるため
- **描画順とスクロール補正で実現する**

## 10. 画面要件

# 10-A. 年表画面

## 10-A-1. 構成
画面は以下の3領域で構成する。

1. **ヘッダー**
2. **年表本体**
3. **補助ナビゲーション**

### デバイス別の役割
- PC / tablet: 閲覧主体
- mobile portrait: 記録主体
- mobile landscape: 記録主体を維持しつつ、軽い閲覧を許容する
- `Selected Period` の保持・同期・センタリングルール自体はデバイスで変えない

### ヘッダーに置くもの
- ロゴまたはタイトル: `Wonder Chronicle`
- 向き切替: `右=現在 / 右=過去`
- 粒度切替: `日 / 週 / 月`
- 起点移動: `today / latest / earliest`
- 記録導線: `追加する` / `編集する`

### モバイル上部の方針
- mobileでは上部常設を **タイトル / その時点の `Selected Period` / `追加する` / `編集する`** に絞る
- `Selected Period` は補助ラベルを増やさず、期間文字列そのものを主表示にする
- 粒度切替は `Selected Period` を押して開くシートにまとめる
- 向き切替と `today / latest / earliest` は `表示` シートにまとめ、常設しない
- `表示` シートを開くときは詳細シートを閉じる

### ヘッダーの方針
- 小さい
- 控えめ
- 年表を圧迫しない
- 設定パネル風にしない

## 10-A-2. 年表本体
- 画面の大半を占める
- 横スクロールを許容する
- 基本は `writing-mode: vertical-rl`
- 列が**右から左へ並ぶ感覚**を作る
- ただし時間の意味は設定で反転可能にする

## 10-A-3. 補助ナビゲーション
- `Selected Period` を小さく表示する
- 例: `2026.04.12 日` / `04/06 - 04/12` / `2026年4月`
- 補助情報は最小限にし、`Selected Period` 以外の状態を増やさない
- mobileでは `表示` ボタンを補助導線として置き、二次操作をシートへまとめる

## 10-B-2. 方針

## 10-B-4. 写真
- まずは1〜3枚
- ローカル画像アップロードでよい
- `URL.createObjectURL` ベースでよい
- 永続化はMVP外

## 15. データ変換責務

### 15.1 変換の流れ
1. `JournalEntry[]` を日付昇順に正規化
2. 粒度に応じて `AggregatedPeriod[]` を生成
3. `AggregatedPeriod[]` を `TimelineColumnViewModel[]` に変換
4. `directionMode` に応じて描画配列を組み立てる
5. `focusedPeriodId` を基準にスクロール位置を決める

### 15.2 責務の分離
- **集約ロジック**: 日/週/月にまとめる
- **代表選定ロジック**: テキストと写真を選ぶ
- **表示変換ロジック**: UI向けテキストや省略表現を作る
- **描画順ロジック**: 向き切替に応じて順序を決める

### 15.3 コンポーネント内でやらないこと
- `TimelineColumn` 内で週集約や月集約をしない
- `TimelineColumn` 内で代表写真を決めない
- `TimelineColumn` 内で日付境界を解釈しない

## 16-A. 粒度切替時の選択ルール

### 16-A-1. 基本方針
- `Selected Period` は `day / week / month` ごとに最後の選択を保持する
- ただし、ある粒度でユーザーが明示的に期間を選んだ場合は、その選択を基準に他粒度の `Selected Period` も整合するよう更新する
- 期間の対応づけは日付文字列ではなく、その期間が含む記録集合を基準に行う

### 16-A-2. 粒度を粗くする場合
- `day -> week`
  選択中の日を含む週を選ぶ
- `week -> month`
  選択中の週を含む月を選ぶ
- `day -> month`
  選択中の日を含む月を選ぶ

### 16-A-3. 粒度を細かくする場合
- `month -> week`
  その月の中で、以前に選んだ週がその月内にあればそれを維持する
- `month -> day`
  その月の中で、以前に選んだ日がその月内にあればそれを維持する
- 以前の選択が新しい親期間に含まれない場合は、その親期間に属する実在記録の週/日へ再設定する
- 空期間は表示しないため、月初や週初が空なら、その期間に属する最初の実在記録へ寄せる

### 16-A-4. 明示選択の優先
- 月で `2026年3月` を選んだあとに週へ切り替えた場合、4月に属する以前の週選択は無効として扱う
- 週で `04/06 - 04/12` を選んだあとに日へ切り替えた場合、その週に含まれる日選択を優先する
- `today / latest / earliest` ジャンプも、明示選択と同様に全粒度の `Selected Period` を同期する

### 16-A-5. スクロール位置
- 粒度切替直後は、その粒度で確定した `Selected Period` が画面内に見えるようにセンタリングする
- 以前の可視中心位置は、粒度切替後の表示基準にはしない

## 17. 写真仕様を固定する

### 17.1 入力方式
- MVPではローカル画像アップロードを許可する
- 表示は `URL.createObjectURL` ベースでよい
- 画面リロードで失われてもよい

### 17.2 代表写真の選定ルール
MVPでは **1日1件制約** があるため、時刻情報は不要とする。

- day表示: その日の `photos[0]`
- week表示: その週に属する日記録を日付昇順に見て、最初に存在する `photos[0]`
- month表示: その月に属する日記録を日付昇順に見て、最初に存在する `photos[0]`
- 写真がなければ `undefined`

将来、1日複数記録を許す場合は `createdAt` または `sortOrder` を導入する。

### 17.3 代表テキストの選定ルール
MVPでは **1日1件制約** があるため、時刻情報は不要とする。

- day表示: その日の記録をそのまま表示
- week表示: その週に属する日記録を日付昇順に見て、最初の記録を代表にする
- month表示: その月に属する日記録を日付昇順に見て、最初の記録を代表にする

将来はAI要約やテーマ抽出へ置き換える。

### 17.4 列内の表示サイズ上限
- 日表示: 列幅の **70%以内**
- 週/月表示: 列幅の **80%以内**
- 高さは列全体の主役になりすぎないよう、**1画像あたり最大 180px 程度** を上限の初期値とする

### 17.5 トリミング方針
- 縦横比は崩さない
- `object-fit: cover` を基本にする
- 表示枠内に収める
- MVPでは中央トリミングでよい

### 17.6 写真なし期間の見せ方
- 画像枠を無理に出さない
- 代わりに小さなプレースホルダ文字を出してもよい
- ただし目立たせない
- 推奨は `写真なし` を薄く添えるか、何も置かない

## 19. CSS / レイアウトの実装ルール

### 19.1 「カードUIにしない」の実装ルール
- 外枠で四辺を強く囲わない
- 背景色で浮かせすぎない
- 強い影を使わない
- 角丸は使っても **2px〜4px程度まで**
- セルごとのボックス感より、**並びの連続感**を優先する

### 19.2 線と余白
- 区切り線は **1px以下の細線** を中心にする
- 巻物型の列間隔は **狭め**
- 社史型を将来入れる場合は **広め**
- 列内余白は十分取るが、外周余白で切れ切れにしない

### 19.3 色と装飾
- 背景は淡色か無彩色を基本にする
- 強いコントラストや装飾は避ける
- 強調はサイズ差と余白で行う

### 19.4 写真の扱い
- 写真を主役にしすぎない
- 文字の流れを切らない位置に置く
- 画像が複数あってもMVP表示は代表1枚でよい

### 19.5 縦書きの基本
- `writing-mode: vertical-rl`
- `text-orientation: mixed`
- 欧文UIラベルは必要に応じて横組みで別管理してよい

## 20. 縦書きレイアウトで破綻しやすい点と回避策

### 20.1 英数字・記号
問題:
- 英数字や `/ - :` などが読みにくくなる

回避:
- 日付ラベルは専用スタイルで調整する
- 必要なら日付だけ横組みも許可する
- まずは全体縦書きの中で、ラベルだけ例外を認める

### 20.2 長文
問題:
- 列幅が細いと縦書き長文が暴れる

回避:
- 行数制限または字数制限を設ける
- 詳細画面に逃がす
- ViewModel段階で省略テキストを作る

### 20.3 写真と縦書きの干渉
問題:
- 画像が列の流れを壊す

回避:
- 画像サイズ上限を設ける
- 画像位置を固定寄りにする
- 画像なしでも成立するレイアウトにする

### 20.4 モバイル横スクロール
問題:
- どちらへ進めばよいか分かりにくい

回避:
- 初回表示位置を明確にする
- ヘッダーに向き状態を明示する
- focusedPeriod を薄く示す

### 29.2 Out of Scope

- AI要約の本実装

- 1日複数記録
- `createdAt` / `sortOrder`

### 29.3 実際のコンポーネント構成
- `App`
  `TimelinePage` 相当の役割を持ち、全体状態、集約、selected period、フォーム表示、詳細表示を束ねる
- `TimelineHeader`
  粒度、向き、ジャンプ、`追加する` / `編集する` の操作を持つ
- `TimelineViewport`
  横スクロール、列描画、ドラッグスクロール、selected period の中央寄せ補正を持つ
- `TimelineColumn`
  日付ヘッダー、写真スロット、3W短冊を持つ縦書き列を描画する
- `EntryForm`
  追加と編集の両方を受け持ち、保存時は 1日1記録制約を検証する
- `EntryDetail`
  期間の代表3Wと、期間に属する日記録一覧を表示する

### 29.4 実際のデータ変換責務
- `normalizeEntries`
  `JournalEntry[]` を日付昇順へ正規化する
- `aggregateEntries`
  粒度に応じて `AggregatedPeriod[]` を生成する
- `toColumnViewModels`
  `AggregatedPeriod[]` を `TimelineColumnViewModel[]` に変換する
- `toRenderedColumns`
  `directionMode` に応じて描画順を変える
- `resolveAnchorPeriodId`
  `today` / `latest` / `earliest` のジャンプ先を確定する

### 29.5 向き切替の実装方針
- データ順は常に古い→新しいで保持している
- `past-right` のときだけ描画配列を反転している
- `selectedPeriodId` を維持し、切替後にその period がビューポート中央へ来るよう `scrollTo` で補正している
- CSS反転は使っていない

### 29.6 縦書きで破綻しやすかった点
- 日付は完全縦組みより横組みタグのほうが読めたため、ラベルだけ横組みにした
- 3Wの英語ラベルは違和感が強かったため、`願い / 感動 / 問い` の短い日本語見出しへ変更し、文字を縦に積む形にした
- 本文と写真の干渉を抑えるため、写真は列上段の固定スロットへ移し、画像は代表1枚かつ最大高 `140px` に制限した
- 週/月でもカード化しないため、背景面より縦線と余白で連続感を作った

### 29.7 サンプルデータでの確認結果
- day / week / month の列数は意図どおり変化した
- 月またぎ週を含むデータで週集約が安定した
- 写真あり / なし の混在でも列構造は崩れなかった
- 長めの `Wonder about` は ViewModel で省略し、詳細パネルへ逃がした
- 同日保存は追加ではなく更新として扱われた
- ドラッグスクロールと列クリックの両立を確認した
- 粒度切替時も `Selected Period` を起点に、対応する週/月/日へ自然に遷移する構成にした

### 29.8 仕様から変更した点
- 技術基盤を `React / Next.js` 候補から `React + Vite + TypeScript` に固定した
- 詳細画面は独立ページではなく、MVPでは右側パネルを基本にした
- 月表示の `summaryTitle` は「その月の最初の日記録の `Wonder at` 抜粋」で実装した
- 年表カラム内の3Wラベルは `Wish / Wonder at / Wonder about` の英語表記ではなく、`願い / 感動 / 問い` の縦見出しで実装した
- 写真は列下部ではなく、日付ヘッダー直下の固定スロットに配置した
- `Focused Period` 表示は `Selected Period` のみを出す構成に整理した
